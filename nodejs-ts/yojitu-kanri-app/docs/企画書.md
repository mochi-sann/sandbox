# 予実管理アプリ企画書

## 1. 企画概要

大規模企業が契約しているソフトウェア、SaaS、ITツールの契約情報、予算、実績、利用状況、従量課金の利用量、更新リスクを一元管理する社内向けウェブアプリを作成する。

部門ごとに散在している契約台帳、請求実績、予算消化、更新期限、利用率、従量課金の消費量を同じ画面で確認できるようにし、コスト最適化とガバナンス強化を支援する。

## 2. 背景と課題

大規模企業では、部署ごとにソフトウェアやITツールを個別契約していることが多く、次の問題が起きやすい。

- 契約情報がスプレッドシート、購買システム、経理データ、各部門の管理表に分散している
- 年間予算に対して、実際にいくら使っているかを即時に把握しづらい
- 従量課金のアプリでは、APIコール数、ストレージ量、処理件数などの利用量と請求額の関係を追いづらい
- 契約更新日が近いツールや、自動更新される契約を見落としやすい
- 同種ツールの重複契約や、利用率の低いライセンスを発見しづらい
- 経営層、IT部門、経理部門、各事業部で見たい粒度が異なる

## 3. 目的

本アプリの目的は、ITツールに関する予算と実績を可視化し、契約更新やコスト見直しの判断を早く正確に行える状態を作ることである。

主な目的は次の通り。

- 契約中のソフトウェアとITツールを一元管理する
- 予算、実績、差異、見込みを部門別・ツール別に可視化する
- 従量課金の利用量、単価、課金見込みを可視化し、急な利用増や予算超過を検知する
- 更新期限、契約リスク、利用率低下を早期に検知する
- 重複契約や未利用ライセンスを洗い出す
- 経営会議や予算会議で使える集計情報を素早く出せるようにする

## 4. 想定ユーザー

| ユーザー | 主な関心 | 代表的な利用シーン |
| --- | --- | --- |
| 情報システム部門 | 契約台帳、更新管理、利用率、セキュリティ評価 | 契約更新前の見直し、重複ツールの整理 |
| 経理・財務部門 | 予算、実績、費目、請求データ | 月次締め、予算差異分析 |
| 各事業部の管理者 | 自部門の契約、利用者、コスト | 部門予算の確認、ライセンス追加申請 |
| 経営層 | 全社コスト、削減余地、リスク | 四半期レビュー、来期予算判断 |
| 購買・法務部門 | 契約条件、更新日、ベンダー情報 | 契約交渉、更新可否判断 |

## 5. 提供価値

### 5.1 コスト管理

全社、部門、ベンダー、ツール単位で予算と実績を比較し、超過や未消化を把握できる。

### 5.2 契約管理

契約期間、更新日、契約金額、支払条件、担当者、契約書リンクを管理し、更新漏れや不要な自動更新を防ぐ。

### 5.3 利用状況の可視化

契約ライセンス数、割当ライセンス数、アクティブユーザー数、利用率を管理し、未利用ライセンスの削減候補を発見する。

### 5.4 従量課金の利用量可視化

APIコール数、ストレージ使用量、処理件数、データ転送量などの従量課金メトリクスを部門別・ツール別に確認し、利用量と請求額の増減を早期に把握できる。

### 5.5 意思決定支援

コスト削減候補、更新優先度、リスク契約を一覧化し、会議や稟議に使いやすい形で情報を整理する。

## 6. MVP機能

初期リリースでは、次の機能を対象とする。

| 機能 | 内容 |
| --- | --- |
| ダッシュボード | 全社予算、実績、差異、更新件数、削減候補を表示 |
| 契約台帳 | ツール名、ベンダー、契約金額、契約期間、更新日、担当部門を管理 |
| 予実管理 | 年度予算、月次実績、見込み、差異を部門別・ツール別に確認 |
| 更新アラート | 30日、60日、90日以内に更新を迎える契約を表示 |
| 利用率管理 | ライセンス数、利用者数、利用率、未利用数を表示 |
| 従量課金管理 | 利用量、課金単位、単価、当月見込み、予算消化率を表示 |
| フィルタ・検索 | 部門、ベンダー、カテゴリ、契約状態、更新期限で絞り込み |
| レポート出力 | CSVまたは画面表示で予実・契約一覧を確認 |

## 7. 将来的な拡張機能

- ワークフロー: 契約更新、解約、追加購入の申請と承認
- 外部連携: 会計システム、購買システム、SSO、IdP、SaaS管理APIとの連携
- 自動アラート: Slack、Teams、メールへの通知
- 監査ログ: 契約情報や予算情報の変更履歴
- 利用量自動取得: 従量課金アプリのAPIから利用量、上限、請求予定額を自動取得
- AI分析: 重複ツール、削減候補、契約交渉ポイントの自動提案
- 権限管理: 全社管理者、部門管理者、閲覧者などのロール管理

## 11. 初期リリース範囲

### 対象に含める

- 手入力またはCSV取り込みによる契約台帳管理
- サンプルデータを使ったダッシュボード
- 予算、実績、差異の可視化
- 契約更新期限の一覧表示
- 利用率と削減候補の表示
- 従量課金アプリの利用量、単価、当月請求見込みの表示

### 対象に含めない

- 会計システムとの自動連携
- SaaS APIからの自動利用状況取得
- 従量課金アプリAPIからの自動利用量取得
- 複雑な承認ワークフロー
- 電子契約システム連携
- 高度な権限管理

## 12. 開発方針

まずはフロントエンド中心のプロトタイプとして、サンプルデータで業務画面の使い勝手を確認する。その後、データ永続化、認証、権限管理、外部連携を段階的に追加する。

推奨する初期構成は次の通り。

- フロントエンド: React + TypeScript + Vite
- UI: レスポンシブ対応の業務ダッシュボード
- データ: 初期はローカルのモックデータ
- 将来のバックエンド: Node.jsまたはHono
- 将来のDB: PostgreSQL

## 14. 主なリスクと対策

| リスク | 対策 |
| --- | --- |
| 契約データが部門ごとに散在している | 初期は主要ツールに絞り、CSVで段階的に取り込む |
| 実績データの粒度が揃わない | 部門、費目、ベンダーの標準分類を定義する |
| 利用率データを取得できないツールがある | 初期は手入力、将来API連携に切り替える |
| 従量課金の利用量単位がツールごとに異なる | APIコール、GB、件数、分数などの課金メトリクスを標準項目として持たせる |
| 利用量の急増を月次締めまで検知できない | 当月見込みと前月比を表示し、しきい値超過をアラート候補にする |
| 権限設計が複雑になる | MVPでは全社管理者と閲覧者を中心に絞る |
| 現場の更新運用に定着しない | 更新アラートと担当者ビューを優先実装する |

## 15. 次に決めること

- MVPで扱う対象ツールの範囲
- 予算管理の単位: 全社、部門、プロジェクト、費目
- 実績データの入力方法: 手入力、CSV、外部システム連携
- 従量課金の利用量単位: APIコール、ストレージ、データ転送量、処理件数、利用時間
- 従量課金データの入力方法: 手入力、CSV、各アプリAPI連携
- 契約更新アラートの通知方法
- 初期画面で最も重視する利用者: 情シス、経理、経営層、部門管理者
